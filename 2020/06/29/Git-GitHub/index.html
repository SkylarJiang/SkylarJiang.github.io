helloGit

Git 是目前世界上最先进的分布式版本控制系统
GitHub 为开源项目免费提供Git存储

分布式和集中式版本控制系统：

集中式：集中存放在中央服务器，必须联网才能工作，效率低。
分布式：每个人的电脑都是一个完整的版本库，协作时只需要把各自的修改推送给对方，无需联网也可以工作；有一台充当“中央服务器”的电脑，用来方便交换大家的修改

创建版本库

版本库，repository，版本库下所有文件都可以被 Git 管理起来，每个文件的修改、删除 Git 都能跟踪

  • 把文件添加到版本库

版本库只能跟踪文本文件的改动，比如TXT文件，网页，所有的程序代码（word 文件不可以）等等；而图片、视频等二进制文件就无法跟踪变化

分支管理

1. 分支是干什么的

如果只有一个分支，大家都在上面进行版本管理，如果不每次提交保存进度就会丢失进度的风险，所以需要提交，但是你不完整代码的提交又会影响他人的工作。分支的作用就是给每个人一个自己进行开发的分支，就可以解决以上矛盾。

2. 基本概念及流程

HEAD 指向当前当前工作区

新建：新建一个 dev 分支就是新建一个 dev 指针，指向master相同的提交

提交：切换到dev分支，head 指向当前 dev 的提交，我们接下来的提交就是在dev分支中进行了。在切换到 dev 分支之后进行的操作都不会对master分支造成影响，切换回master就能看到master原来的样子了

合并：把master指向当前的提交点（从原先提交指向dev当前提交点，链表增加一个数），就完成了合并，对于master分支而言，只是多了一个提交

操作：切换到master分支之后把dev分支的内容合并过来（因为是在master上进行操作的，所以是将 dev 的内容搞过来，对dev 分支怎么会造成影响呢）

合并分支有多种情况。

删除：合并分支之后可以删除 dev 分支，也就是删除 dev 指针。那么dev以前的提交去哪了，直接消失了？

远程分支

  • git clone：gitclone 是没办法拿到dev分支的内容的，clone后面只能跟仓库名
  • git remote：远程主机相关操作
  • git branch
  • git pull （= fetch + merge）
  • git push (推送本地 commit)
  • git 合并两个远程分支：有一天我想合并一个开源项目的两个分支，但是我只有pull权限

那么请问 pull 方法拉取分支进行操作和 让本地分支建立和远程分支的连接 有什么不同嘞？不都是拿过来然后 push 新的commit 就能更新远程分支了？那可能是因为权限问题吧，建立连接了是可以实现同步了，但是有些别人的库你没有更改的权限，只能pull过来自己玩一下

3. 解决冲突

3.1 合并方法

  • 快速合并：master没有改变，只有dev分支改变了，可以直接改变master执行，进行快速合并（合并纯属是master在改变，无需商量）
  • 手动合并：如果需要解决冲突那就不能用快速合并了，——在(master|Merging)分支
    • 如果直接进行合并会报错让我们手动解决冲突再提交，可以通过 git status 查看
    • 此时冲突文件中会标记出不同分支的冲突的内容，我们可以在这个文件中手动进行修改，重新 add & commit
    • 合并完成，分支变回 master

3.2 查看合并记录

4. 分支管理策略

  • 禁用快速合并：Git 默认会尽量采取 fast forward 模式进行合并，但是这种模式下删除分支之后会丢掉分支信息。如果禁用快速合并，Git 就会在merge的时候生成一个新的commit，这样就可以在分支历史中就可以看出分支信息

  • 分支策略

    工作中一般不直接在 master 分支上操作，仅用来发布新版本
    工作团队就在 dev 分支上工作，从dev分支中再分出属于自己的分支

5. bug分支

  • 通过新建分支来修复 bug 再进行合并
  • 此时如果还有别的分支上的工作还在进行不能先提交，但是不提交又会有冲突那咋办。可以同过stash功能来把当前工作现场先保存，再新建别的分支修复 bug。

6. Feature 分支

  • 添加新功能
  • 添加新功能的时候先再新建的 feature 分支上开发，完成后再合并
